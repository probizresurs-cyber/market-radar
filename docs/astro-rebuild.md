# Перенос сайта на Astro + Оптимизация

Инструмент MarketRadar: берёт живой сайт (обычно на конструкторе — Tilda,
WordPress и т.п.), пересобирает его как статический Astro-проект **с
сохранением дизайна 1:1**, исправляет технический SEO-внутряк и — отдельным
шагом — оптимизирует скорость загрузки. Это техническая база двух услуг из
пилотных КП: «Перенос сайта на Astro» и «Оптимизация».

- **Страница:** `marketradar24.ru/astro-rebuild` — публичная, вход не нужен.
- **Результат шерится ссылкой:** `…/astro-rebuild?id=<id>` — открывший видит
  готовый результат (живой сайт, отчёты, исходники) и может пересобрать свой.
- **Живой пересобранный сайт:** `…/api/site-preview/<id>` — публичный URL,
  можно отправлять клиенту напрямую.

---

## Как пользоваться (коротко)

1. Вставить адрес сайта → **«Пересобрать»** (~20–40 сек).
2. Посмотреть табы: **Живой сайт** (превью в странице), **Оптимизация**
   (диагностика), **Исходники** (дерево файлов Astro-проекта).
3. На табе «Оптимизация» → **«Применить оптимизацию»** (1–2 мин).
4. **«Замерить»** — честное сравнение «было → стало» через Google PageSpeed.
5. **«Скачать .zip»** — готовый проект: `npm install && npm run build`.

---

## Этап 1 — Перенос (быстрый, без скачивания ассетов)

Философия: **дизайн не трогаем вообще**. Никакой AI-генерации вёрстки —
берём оригинальный HTML как есть и хирургически правим только внутряк.
AI (Claude Sonnet) используется в одном маленьком запросе: сгенерировать
текст meta description и Schema.org из реального контента страницы
(~3 тыс. токенов, ≈1–1.5 ₽ за пересборку).

Что делает по шагам:

1. **Скрейп** — скачиваем страницу (до 3 МБ), собираем метаданные: заголовки,
   картинки, навигацию, соцсети, стек, наличие robots/sitemap.
2. **DOM-хирургия** (правки, не меняющие вид):
   - все ссылки на CSS/картинки/видео → абсолютные (дизайн грузится 1:1);
   - материализация ленивой загрузки конструктора: реальные картинки из
     `data-original` прописываются в вёрстку — контент виден без JS
     (важно для поисковиков и нейросетей);
   - `meta description`, `canonical`, `viewport`, Open Graph, Schema.org
     (JSON-LD) — добавляются, только если их не было;
   - один смысловой `H1`: если нет ни одного — первый заголовок из
     `main/article` повышается до H1 с сохранением классов (вид не меняется);
   - `alt` у всех изображений без описания;
   - `lang/dir/class` оригинального `<html>` переносятся в Astro-Layout.
3. **Диагностика производительности** (только находит, не правит): что
   тормозит страницу и что исправит этап оптимизации.
4. **Сборка Astro-проекта**: `Layout.astro` (голова) + `index.astro` (тело)
   через `set:html`, `astro.config.mjs` c обязательным `site` (без него
   падает `@astrojs/sitemap`), `package.json`, `robots.txt`, README.
5. Снапшот сохраняется в БД — результат переживает перезагрузку и доступен
   по публичной ссылке.

Что **сознательно не** трогаем: скрипты конструктора (на них слайдеры и
формы), видео (слишком тяжёлые — остаются на оригинальном домене).

## Этап 2 — Оптимизация (отдельная услуга, кнопка на табе)

Запускается от «чистого переноса» (baseHtml), поэтому повторные запуски не
наслаивают правки. Дизайн по-прежнему не меняется. Шаги:

1. **Ленивая загрузка `<img>`** ниже первого экрана (`loading="lazy"` +
   `decoding="async"`; первые 2 картинки — сразу, чтобы не портить LCP).
2. **Ленивые фоны** — ключевой фикс. После переноса фоновые картинки
   конструктора прописаны в вёрстке напрямую и грузились бы **все разом**
   (у Tilda-сайтов это 90–100 изображений — главный тормоз). Оптимизация
   возвращает подгрузку по прокрутке: 15-строчный инлайн-загрузчик на
   IntersectionObserver (запас 400px), первые 4 блока — мгновенно.
3. **Уборка двойной загрузки**: атрибуты `data-original`/`data-lazy-src`
   вычищаются — иначе скрипт конструктора качает те же картинки второй раз
   со своего CDN.
4. **Ленивые iframe** (карты, встройки).
5. **Перенос ассетов к себе**: картинки, CSS и шрифты (включая шрифты из
   CSS-файлов) скачиваются на наш сервер и отдаются с него с вечным кэшем.
   Сайт перестаёт зависеть от CDN конструктора; заодно чинится 403 на
   шрифтах Tilda (реферер-защита). Лимиты: 25 МБ/файл, 100 МБ/пересборку,
   150 файлов, SSRF-проверка каждого домена.
6. **Сжатие картинок**: JPEG/PNG → WebP (поворот по EXIF, ресайз до 1920px,
   качество 80). Если WebP вышел больше оригинала — честно остаётся оригинал.
7. **preconnect** к реально оставшимся внешним доменам (скрипты
   конструктора) + **preload** первого изображения.

После оптимизации найденные проблемы помечаются «Исправлено», а старый замер
скорости сбрасывается — мерить нужно заново, честно.

## Замер «было → стало»

Кнопка «Замерить» гоняет **Google PageSpeed (Lighthouse, mobile)** по
оригинальному сайту и по нашему превью параллельно и показывает бок о бок:
Performance-балл, LCP, FCP, TBT, CLS, вес страницы. Это реальные цифры
Google, не наши оценки; замер сохраняется в снапшот и виден по публичной
ссылке. Если замер одной из сторон упал — показывается `—` и причина.

## Реальный кейс: sozdavay.art (Tilda), 20.07.2026

Найдено при переносе (SEO): нет H1, alt у 0 из 7 картинок, нет Schema.org.
Исправлено переносом: 4 правки (абсолютные ссылки, материализация 99
картинок/фонов, Schema.org, alt).

Применено оптимизацией (7 правок):

- 94 фоновых изображения → подгрузка по прокрутке;
- убрана двойная загрузка 99 картинок скриптом Tilda;
- перенесено к себе 150 ассетов (11 CSS);
- **сжато 124 изображения в WebP: 39.9 МБ → 15.5 МБ (−61%)**;
- ленивые `<img>`, preconnect, preload.

Замер Lighthouse (mobile), оригинал → после оптимизации:

| Метрика | Оригинал | После | Δ |
|---|---|---|---|
| Performance | 44 | **58** | **+14 баллов** |
| LCP (крупный контент) | 36.1 с | 20.1 с | −44% |
| FCP (первая отрисовка) | 6.5 с | 4.0 с | −38% |
| TBT (блокировка) | 466 мс | 63 мс | −86% |
| Вес страницы | 14.3 МБ | 8.2 МБ | −43% |

Что осталось и почему (честные рекомендации, авто-фикс невозможен):

- **HTML 1256 КБ** — конструктор кладёт всю вёрстку и стили инлайн; лечится
  только чистой пересборкой вёрстки (ручная работа);
- **34 внешних скрипта + jQuery** — на них слайдеры/формы Tilda; их
  сокращение — ручная доработка (это главный потолок Lighthouse-балла);
- **12 CSS-файлов** — объединение возможно при ручной доработке.

Важно: превью отдаётся с нашего VPS без CDN. После деплоя проекта на
нормальный статический хостинг (CDN + brotli + кэш) цифры будут ещё лучше.

## Скачанный проект

Zip собирается на сервере и включает бинарные ассеты:

```
package.json          # astro + @astrojs/sitemap
astro.config.mjs      # site: <origin> (обязателен для sitemap)
tsconfig.json
src/layouts/Layout.astro
src/pages/index.astro
public/robots.txt
public/assets/        # локализованные картинки (WebP), CSS, шрифты
README.md
```

Проверка: `npm install && npm run build` → папка `dist/`, просмотр —
`npm run preview`.

## API (для интеграций)

| Метод | Путь | Что делает |
|---|---|---|
| POST | `/api/rebuild-astro` `{url}` | Этап 1: перенос + диагностика. IP-лимит 15/день |
| POST | `/api/rebuild-astro/optimize` `{id}` | Этап 2: оптимизация. IP-лимит 15/день |
| POST | `/api/rebuild-astro/compare` `{id}` | Замер PSI обеих версий. IP-лимит 10/день |
| GET | `/api/rebuild-astro/<id>` | Сохранённый результат (гидрация страницы) |
| GET | `/api/rebuild-astro/<id>/zip` | Готовый проект архивом |
| GET | `/api/site-preview/<id>` | Живой пересобранный сайт (публично) |
| GET | `/api/rebuild-asset/<id>/<file>` | Локализованные ассеты (immutable-кэш) |

Хранение: снапшоты — Postgres (`astro_rebuilds`), бинарные ассеты —
`/var/lib/marketradar/astro-rebuild-assets/<id>/` (env
`ASTRO_REBUILD_ASSETS_DIR`), автоуборка каталогов старше 14 дней.
Опционально: `PAGESPEED_API_KEY` снимает квоту Google на замеры.

## Известные ограничения

- Пересобирается **главная страница** (один URL); многостраничный обход —
  в планах.
- Анимации «появления при скролле» и слайдеры конструктора зависят от его
  JS — в статике часть эффектов статична (контент не теряется).
- Видео не переносятся (остаются на оригинальном домене).
- Балл Lighthouse упирается в скрипты конструктора — их диета («лёгкая
  версия» без JS Tilda) возможна как следующий этап, но ломает
  слайдеры/формы, поэтому только как осознанная опция.
